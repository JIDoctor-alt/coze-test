## 术语对照表（小白的冰箱贴）

| 术语 | 人话解释 |
|------|----------|
| `@Transactional` | 银行转账，要么全成功要么全回滚（要么全部数据库操作都完成，要么一步失败全部撤销） |
| `@Cacheable` | 冰箱贴，常用数据贴冰箱不用每次翻（把查询结果缓存起来，下次直接用） |
| `Mapper` | 菜谱，告诉数据库按什么方子找数据 |
| `Feign调用` | 打电话给另一个部门帮忙干活 |
| `ServiceImpl` | 业务逻辑的"厨师长"，负责炒菜（处理业务） |
| `Dao` | 数据库操作的"仓库管理员"，负责存取数据 |
| `Activiti` | 工作流引擎，就像流水线加工，任务沿着固定流程走 |
| `乐观锁` | 相信大家不会抢，先干活最后检查 |
| `悲观锁` | 先锁门再干活，别人等着 |

#### 1.1 `queryMbdeclarePhysicalInfoList()` - 查询申报体检信息列表

**📌 一句话解释：** 就像医院的挂号系统，根据各种条件筛选出需要体检的申报单

**参数说明：**
| 参数 | 类型 | 说明 |
|------|------|------|
| selectVipMbdeclareInfoVo | QueryMbInfoVo | 包含分页、筛选条件（申报状态、体检状态、单位编码等） |

**返回值：** `ResultPage<VipMbdeclareInfoDto>` - 分页的申报信息列表

**核心逻辑拆解：**
```
1️⃣ 解析版本代码 → 如果包含@符号，拆分出时间范围
   例如："BJ2024@2024-01-01@2024-12-31" → 版本BJ，时间范围起始结束
   
2️⃣ 设置默认体检状态 → 如果没传体检状态，默认设为9（未知）
   
3️⃣ 判断数据范围 → 根据flag判断是哪个城市的申报数据
   - "BJ" = 宝鸡
   - "SL" = 商洛
   - 其他 = 默认配置
   
4️⃣ 根据是否有工作流配置选择查询方式：
   - 有工作流 → 使用activitiDao查询（通过工作流引擎查）
   - 无工作流 → 使用vipMbdeclareInfoDao直接查数据库
   
5️⃣ 宝鸡数据需要额外处理 → 补充人员身份、联系方式等信息
```

**💡 小白易懵点：**
> 这个方法用了策略模式，根据不同城市的配置（UnitConfig）选择不同的数据访问层。就像同一个小区，有多个入口（直接走大门/走侧门/走地下车库），但最后都到同一栋楼。

#### 1.2 `saveMbdeclareInfo()` - 保存申报信息

**📌 一句话解释：** 把患者的申报材料像装进档案袋一样存起来，并启动审批流程

**参数说明：**
| 参数 | 类型 | 说明 |
|------|------|------|
| insertVipMbdeclareInfoVo | InsertVipMbdeclareInfoVo | 包含患者信息、病种、联系方式等 |
| isFirst | Boolean | 是否首次申报（区分新增和修改） |

**返回值：** `ApiResponse<String>` - 返回申报ID

**核心逻辑拆解：**
```
1️⃣ 保存或更新主表 → vip_mbdeclare_info（申报主表）
   - 首次申报 → 新增记录
   - 修改申报 → 更新记录，状态改为"待初审核"
   
2️⃣ 保存申报文件 → vip_mbdeclare_file（申报附件表）
   - 身份证照片、病例资料、处方单等
   - 文件路径存到数据库，文件实际存在FTP服务器
   
3️⃣ 启动工作流 → Activiti引擎开始审批流程
   - 创建流程实例 → 分配给初审人员
   
4️⃣ 保存手机变更记录 → 如果用户修改了手机号，记录变更历史
   
5️⃣ 发送短信通知 → 告诉用户"您的申报已提交，请等待审核"
```

**异常处理：**
- 文件上传失败 → 事务回滚，提示用户重新上传
- 工作流启动失败 → 记录日志，人工介入
- 手机号被占用 → 提示用户确认

**💡 小白易懵点：**
> 这里的"保存"不是简单的INSERT，而是包含工作流启动的复杂操作。就像寄快递，不是把包裹放到驿站就完了，而是要扫码跟踪、分配快递员、开始派送。

#### 1.3 `getVipMbdeclareInfoById()` - 根据ID获取申报详情

**📌 一句话解释：** 翻阅某个具体患者的完整申报档案

**参数说明：**
| 参数 | 类型 | 说明 |
|------|------|------|
| id | String | 申报记录ID |

**返回值：** `VipMbdeclareInfo` - 完整的申报信息对象

**核心逻辑拆解：**
```
1️⃣ 先查缓存 → Redis里有没有这份申报的详细信息
   - 缓存key格式："ya:allKey:" + 申报ID
   - 缓存时间：通常1小时
   
2️⃣ 缓存没有 → 查数据库
   - SELECT * FROM vip_mbdeclare_info WHERE id = ?
   
3️⃣ 补充文件列表 → 查询该申报的所有附件
   - 身份证、病历、处方、审批文件等
   
4️⃣ 补充体检信息 → 如果已分配体检中心，查询体检详情
   
5️⃣ 写回缓存 → 下次查询直接读缓存
```

**@Cacheable 解读：**
> 把查询结果贴"冰箱贴"，1小时内再次查询直接从冰箱拿，不用去档案室翻。

#### 2.1 `updateInfoDZ()` - 初审审批（德州版本）

**📌 一句话解释：** 审批人员看完材料后，决定"通过/驳回/补充材料"

**参数说明：**
| 参数 | 类型 | 说明 |
|------|------|------|
| insertVipMbdeclareInfoVo | InsertVipMbdeclareInfoVo | 审批结果和申报信息 |
| flag | String | 地区标识 |

**返回值：** void（直接更新数据库）

**核心逻辑拆解：**
```
1️⃣ 获取当前审批人信息 → 从Session获取用户ID、姓名、账号

2️⃣ 判断审批结果：
   ┌─────────────────────────────────────────┐
   │ 审批状态码对比ApprovalStatusEnum        │
   ├─────────────────────────────────────────┤
   │ PASS（通过=2）→ 初审通过                │
   │   - 设置applystatus为"初审通过"         │
   │   - 如果没分配体检中心 → 设置体检状态   │
   │   - 创建/更新会员信息                   │
   │   - 记录初审人员信息                    │
   ├─────────────────────────────────────────┤
   │ NOPASS（驳回=3）→ 初审驳回              │
   │   - 设置applystatus为"初审不通过"        │
   │   - 设置endstatus为"已结案"（流程结束） │
   ├─────────────────────────────────────────┤
   │ ZLBQ（补充材料=4）→ 需要补正            │
   │   - 设置applystatus为"补充材料"          │
   └─────────────────────────────────────────┘

3️⃣ 达州特殊处理：
   - 初审驳回 → 发送短信通知申请人
   - 短信内容："您申报的门诊慢性病资料审核未通过..."

4️⃣ 记录审批日志 → vip_mbdeclare_approval表
   - 谁审的、什么时候审的、结果是什么

5️⃣ 完成工作流任务 → taskService.complete()
   - 推进到下一个审批节点（如复审/体检分配）
```

**异常处理：**
- 手机号被占用 → "手机号码被他人占用，请确认"
- 备注超长 → "备注信息超过200，请重新输入"

**@Transactional 解读：**
> 这个方法是一个完整的"审批事务"，要么全部更新成功，要么全部回滚。想象银行转账：更新申报状态、更新会员信息、记录审批日志、发送短信，要么全做要么全不做。

**💡 小白易懵点：**
> 审批状态码用的是字符串比较 `approvalStatus.compareTo(String.valueOf(ApprovalStatusEnum.PASS.getIntValue())) == 0`，而不是直接的 `==` 或 `equals`。这是因为前端传来的可能是"2"字符串，需要转成整数的2再比较。

**📌 一句话解释：** 专家看完补充材料后的最终裁决

**返回值：** `ApiResponse<String>`

5️⃣ 记录审批日志 → 包含专家意见、病种编码等

### 方法解析

#### 3.1 `VIPPayMoney()` - 会员卡消费支付

**📌 一句话解释：** 刷卡买药，系统自动扣钱并打印小票

**参数说明：**
| 参数 | 类型 | 说明 |
|------|------|------|
| request | XMLTable<VIPPayMoneyRequestVo> | 包含卡号、密码、消费金额、商品列表等 |

**返回值：** `XMLTable<VIPPayMoneyResponseVo>` - 返回订单号、流水号、消费金额

**核心逻辑拆解：**
```
1️⃣ 参数校验
   ┌─────────────────────────────────────────┐
   │ 校验规则：                              │
   │ - 卡号和账号不能同时为空               │
   │ - 消费金额必须 > 0                     │
   │ - 密码不能为空（后台支付除外）         │
   └─────────────────────────────────────────┘

2️⃣ 账户合法性检查
   - 根据卡号/账号查询账户
   - 检查账户状态（4=冻结，不能消费）
   - 检查账户锁定状态（lockflag非0=锁定）
   - 检查账户有效期（过期不能消费）
   - 检查账户类型（保额报销账户不能直接消费）

3️⃣ 密码验证
   - 加密后比对数据库存储的MD5值
   - 错误5次会锁定账户（调用accountpasserr）
   - 检查是否为初始密码（初始密码不能消费）

4️⃣ 商户校验
   - 查询商户类型（002=药店等）
   - 记录消费商户信息

5️⃣ 扣款操作（如果是账户支付）
   ┌─────────────────────────────────────────┐
   │ 悲观锁：SELECT ... FOR UPDATE           │
   │ - 先把账户金额那行锁住                 │
   │ - 检查余额是否足够                     │
   │ - 余额 -= 消费金额                     │
   │ - UPDATE账户金额                       │
   └─────────────────────────────────────────┘

6️⃣ 记录资金流水 → vip_moneyflow
   - 流水类型：MINUS（支出）
   - 记录消费前余额、消费后余额
   - 记录商户信息、操作员、订单号

7️⃣ 发送消费短信
   - "您尾号1234的卡消费50.00元，余额1000.00元"

8️⃣ 记录商品明细 → vip_productflow
   - 每个商品一条记录
   - 包含商品编码、名称、单价、数量、金额
```

**异常处理：**
- 卡号不存在 → "卡号不存在"
- 密码错误 → "密码错误"（累计错误次数）
- 账户冻结 → "账户已冻结，无法消费"
- 余额不足 → "余额不足，当前余额XX元"
- 初始密码 → "当前密码是初始密码，不能进行消费"

**@Transactional 解读：**
> 扣款+记录流水+发短信，必须同时成功或同时失败。就像ATM取钱：吐钞和扣账必须一致，不能出现钱出来了但账户没扣钱的情况。

**💡 小白易懵点：**
> 商户类型merchanttype用于区分不同类型的消费场所（药店、医院、超市等），不同类型可能享受不同的结算政策和优惠。

## 4. VipAccountmoneyServiceImpl

**账户资金服务 - 账户余额的"保险箱"，精确记录每一分钱**

### 核心依赖
```
- VipAccountmoneyDao: 账户资金数据访问
- RedisUtil: Redis缓存（用于分布式锁）
```

### 方法解析

---

#### 4.1 `getVipAccountmoneyForupdate()` - 获取账户资金（加锁）

**📌 一句话解释：** 去银行保险库取钱，先把保险柜锁上防止别人同时操作

**参数说明：**
| 参数 | 类型 | 说明 |
|------|------|------|
| vipAccountmoney | VipAccountmoney | 查询条件（通常传账户ID） |

**返回值：** `VipAccountmoney` - 账户资金详情（含当前余额）

**核心逻辑拆解：**
```sql
-- 这个方法背后执行的是：
SELECT * FROM vip_accountmoney 
WHERE id = ? 
FOR UPDATE
```

**悲观锁解读：**
> `FOR UPDATE` 就像在数据库里"先锁门再干活"：
> - 其他事务想查同一行数据？→ 排队等着
> - 等当前事务提交/回滚？→ 才能查到最新数据
> - 防止并发扣款时出现"超卖"问题

**💡 小白易懵点：**
> 为什么不能直接查？想象双十一秒杀，100件商品剩最后1件，两个人同时下单：
> - 不加锁：都查到有货，都下单，超卖！
> - 加锁：第一个查到并锁定，第二个等第一个完事后再查，没货了，正确！

---

#### 4.2 `updateByCardno()` - 更新账户余额

**📌 一句话解释：** 从账户里加钱或扣钱，并记录修改人

**参数说明：**
| 参数 | 类型 | 说明 |
|------|------|------|
| vipAccountmoney | VipAccountmoney | 包含账户ID和变更金额（正数加、负数扣） |

**返回值：** int - 影响行数（通常为1）

**核心逻辑拆解：**
```sql
-- 这个方法背后执行的是：
UPDATE vip_accountmoney 
SET money = money + #{vipAccountmoney.money},
    modifier = #{vipAccountmoney.modifier},
    modifytime = NOW()
WHERE id = #{vipAccountmoney.id}
```

**💡 小白易懵点：**
> 用 `money = money + ?` 而不是 `money = ?`，是因为并发场景下：
> - 场景：余额100元，两笔20元扣款同时到
> - 错误做法：事务A set money=80，事务B set money=80 → 余额80（实际应为60）
> - 正确做法：事务A set money=money-20，事务B set money=money-20 → 余额60
