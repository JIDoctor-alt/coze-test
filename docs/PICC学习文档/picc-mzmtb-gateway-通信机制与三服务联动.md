## 二、HTTP通信机制详解 🚀

### 2.1 什么是HTTP调用？

> 🌰 **栗子**：HTTP就像外卖平台和餐厅之间的下单系统。你（客户端）告诉平台要吃什么，平台把你的需求"打包"发送给餐厅（服务器），餐厅做好后再把食物"打包"送回来。

### 2.2 前台服务的三种"传话方式"

前台服务像个万能翻译官，有三种方式和后端"传话"：

#### 方式一：HttpForwardUtil - 智能传令兵

```java
// 核心代码逻辑（已脱敏）
public class HttpForwardUtil {
    
    /**
     * 像传令兵一样，把前线的消息送到指挥部
     * 
     * 🌰 栗子：
     * 你在前台点了餐，传令兵把你的订单送到后厨
     * 后厨做好后，传令兵再把餐送回来
     */
    public static String forward(String url, Map<String, Object> params) {
        // 1. 把参数打包成JSON
        String jsonBody = JSON.toJSONString(params);
        
        // 2. 发送HTTP POST请求
        HttpResponse response = httpClient.post(url, jsonBody);
        
        // 3. 拿到响应结果
        return response.getBody();
    }
}
```

**适用场景**：前台服务向业务服务或权限服务发送请求

#### 方式二：WebService调用 - 外交信使

> 🌰 **栗子**：WebService就像两个国家之间的外交信使。两个系统可能说着不同的"方言"，但外交信使会把消息翻译成双方都能听懂的"外交密码"。

```java
// HmLinkWebserviceBL.java（医保对接专用）
public class HmLinkWebserviceBL {
    
    /**
     * 与医保核心系统通信
     * 
     * 场景说明：
     * 当患者需要核实医保信息时
     * 前台服务会派"外交信使"去问医保系统
     */
    public String callMedicalInsurance(String xmlRequest) {
        // 1. 把请求打包成XML格式（国际标准格式）
        String formattedXml = formatToMedicalStandard(xmlRequest);
        
        // 2. 派信使出发（SOAP协议调用）
        String xmlResponse = webServiceClient.invoke(formattedXml);
        
        // 3. 解析医保系统的回复
        return parseResponse(xmlResponse);
    }
}
```

#### 方式三：Restful API - 标准邮件系统

> 🌰 **栗子**：Restful就像标准的邮件系统。每个请求都是一封信，有收件人地址（URL）、信封上写清楚要干什么（GET/POST/PUT/DELETE）。

```java
// HmLinkRestfulBL.java
public class HmLinkRestfulBL {
    
    /**
     * 与医保系统Restful接口通信
     * 
     * 🌰 栗子：
     * GET 请求 = 明信片（只是查询）
     * POST 请求 = 挂号信（发送重要数据）
     * PUT 请求 = 修改信件（更新数据）
     * DELETE 请求 = 撤信（删除数据）
     */
    public ResponseEntity<String> query(String endpoint, Map<String, String> params) {
        // 使用RestTemplate发送标准HTTP请求
        return restTemplate.getForEntity(
            "http://medical-insurance-api" + endpoint,
            params
        );
    }
}
```

### 2.3 三种方式对比表 📊

| 方式 | 比喻 | 特点 | 适用场景 |
|------|------|------|----------|
| **HttpForwardUtil** | 传令兵 | 灵活、快速 | 内部服务间通信 |
| **WebService** | 外交信使 | 标准化、跨语言 | 与医保系统对接 |
| **Restful** | 标准邮件 | 清晰、规范 | 开放API对接 |

## 三、Token与Session的跨服务传递 🎫

### 3.1 什么是Token？

> 🌰 **栗子**：Token就像游乐园的通行证。你在门口买票后得到一张通行证，拿着这张证可以进入任何一个项目区，不用每个项目都重新买票。

### 3.2 什么是Session？

> 🌰 **栗子**：Session就像酒店的房卡系统。你在前台办理入住后拿到房卡，这张房卡在整个酒店都能用——开门、开电梯、去健身房、吃早餐。Redis就是那个"中央制卡中心"，让三栋楼（三个服务）都能识别同一张房卡。

### 3.4 后续请求中的Token传递

```java
// 前台服务收到带Token的请求后...
public String handleRequest(HttpRequest request) {
    
    // 1. 从请求头取出Token（就像服务员接过你的房卡）
    String token = request.getHeader("Authorization");
    
    // 2. 问问Redis：这个Token有效吗？
    UserInfo user = redis.get("token:" + token);
    
    // 3. Token有效，继续处理业务
    if (user != null) {
        // 4. 带着Token继续向后传递（像传令兵带着通行证）
        return businessService.process(
            request, 
            "from_user:" + user.getUserId()  // 标记是谁的请求
        );
    }
    
    return "Token无效，请重新登录";
}
```

## 七、核心概念通俗解释 🎓

### 7.1 HTTP转发 - 像传令兵

> **通俗解释**：传令兵负责把前线的消息送到指挥部，再把指挥官的命令带回来。他不自己做决定，只是跑来跑去传递信息。
>
> **对应到系统**：前台服务就是那个传令兵，把用户的请求送到业务服务或权限服务，再把结果带回来。

### 7.2 Token - 像工作证

> **通俗解释**：工作证是你在公司里的身份证明。进每个办公室都要出示，但不用每次进门都重新证明"你是谁"。
>
> **对应到系统**：Token就是用户的"数字工作证"，一次登录，长期有效。

### 7.3 Session共享 - 像连锁酒店会员

> **通俗解释**：你在A城市的连锁酒店办了会员卡，积分、折扣在B城市的同品牌酒店也能用，因为他们的会员系统是连通的。
>
> **对应到系统**：Redis就是那个"中央会员系统"，让三个服务都能识别同一个用户的登录状态。

### 7.4 WebService - 像外交信使

> **通俗解释**：两个说着不同语言的国家要沟通，就派外交信使。信使把本国的话翻译成"外交密码"，对方信使再翻译成本国语言。
>
> **对应到系统**：和医保系统对接时，用WebService把请求包装成标准XML格式，医保系统用同样格式回复。

### 7.5 为什么要分层？ - 像去医院看病

> **通俗解释**：
> - **挂号处**（前台服务）：接待你，检查你是不是挂对了科，但不给你看病
> - **门诊医生**（业务服务）：真正给你看病的
> - **收费处**（权限服务）：验证你挂的号真不真、有没有付款资格
>
> **对应到系统**：分层让每个服务各司其职，专业的人做专业的事。

## 八、常见问题解答 💡

### Q1：为什么要用三个服务？一个服务搞定不行吗？

**比喻**：就像一家大公司为什么要分前台接待部、人力资源部、业务部门？
- 分工明确，每个部门做自己专业的事
- 一个部门崩了，其他部门还能运转
- 方便找人负责，出问题知道找谁

### Q2：Redis挂了会怎样？

**后果**：
- 用户登录后拿到的Token就验证不了了
- 已经登录的用户不受影响（浏览器里的Token还在）
- 新用户无法登录
- **解决方案**：Redis一般做集群，主备自动切换

### Q3：前台服务怎么知道该找谁？

**机制**：
- 根据请求的URL路径判断
- `/api/auth/*` → 权限服务
- `/api/biz/*` → 业务服务
- `/api/medical/*` → 医保对接

### Q4：用户请求慢怎么排查？

**排查步骤**：
1. 看前台服务日志 → 请求收到了吗？
2. 看权限服务日志 → Token验证通过了吗？
3. 看业务服务日志 → SQL执行了吗？
4. 看Redis → 缓存有吗？Redis响应正常吗？
5. 看GaussDB → 数据库连接正常吗？SQL执行时间长吗？

## 十、总结 🎯

### 核心要点回顾

1. **前台服务是"守门员"**：所有请求都从这进，负责分配到正确的地方
2. **权限服务是"安检"**：验证身份，发放Token
3. **业务服务是"干活的"**：处理真正的业务逻辑
4. **Redis是"信息中转站"**：让三个服务之间能"互相认识"
5. **GaussDB是"档案库"**：永久保存所有重要数据

### 一句话理解

> 想象你去政务中心办事：
> - **前台服务** = 取号机 + 引导员
> - **权限服务** = 身份证验证
> - **业务服务** = 窗口办事员
> - **Redis** = 叫号屏（实时同步你在哪一步）
> - **GaussDB** = 档案室（永久保存你的材料）
