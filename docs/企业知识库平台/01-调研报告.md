## 一、部署方案对比

### 1.1 阿里云服务方案

| 方案 | 服务名称 | 特点 | 适用场景 |
|------|----------|------|----------|
| **百炼知识库** | Model Studio | 开箱即用、托管服务、按量付费 | 快速上线、中小规模 |
| **表格存储知识库** | Tablestore + OSS | Serverless架构、多租户支持 | 大规模企业级应用 |
| **自建 RAGFlow** | ECS + 容器 | 完全可控、可二开 | 定制化需求强 |

### 1.2 阿里云百炼知识库

**优势**：
- 无需部署，开箱即用
- 支持多种向量模型（text-embedding-v3/v4）
- 内置 RAG 能力，支持混合检索
- 支持 GraphRAG 图谱增强

**限制**：
- 商业化计费，规格费用 + 模型调用费用
- 定制化能力有限
- 数据存储在阿里云

### 1.3 自建 RAGFlow 方案

**硬件要求**：
```
CPU: ≥4核
内存: ≥16GB（推荐32GB）
硬盘: ≥50GB SSD
Docker: ≥24.0.0
GPU: 可选，用于加速 Embedding 和 MinerU
```

**推荐部署架构**：
```
阿里云 ECS（8核32GB）+ 
Docker Compose +
阿里云 RDS（MySQL）+
阿里云 OSS（文件存储）+
阿里云 Elasticsearch（向量存储）
```

## 二、RAGFlow vs RAGFlow-Plus 对比

### 2.1 功能对比

| 功能 | RAGFlow | RAGFlow-Plus |
|------|---------|--------------|
| 后台管理系统 | ❌ 无 | ✅ 多维度权限体系 |
| 文档解析引擎 | DeepDoc | **MinerU**（精度更高） |
| 中文排版优化 | 一般 | ✅ 专项优化 |
| 图文联动输出 | ❌ | ✅ 自动关联图片 |
| 文档撰写模式 | ❌ | ✅ 知识库辅助创作 |
| 部署方式 | Docker | Docker + 轻量化 |

### 2.2 MinerU vs DeepDoc

| 指标 | DeepDoc | MinerU |
|------|---------|--------|
| 解析精度 | 70-80% | **82-90%+** |
| 格式支持 | PDF/图片 | PDF/Word/PPT/Excel/图片 |
| 中文优化 | 一般 | ✅ 专项优化 |
| GPU 需求 | 可选 | 可选（纯CPU也支持） |
| 显存要求 | 6GB+ | 6GB+（hybrid模式） |

### 2.3 选择建议

| 场景 | 推荐方案 |
|------|----------|
| 快速验证、无GPU | RAGFlow（CPU模式） |
| 生产环境、中文为主 | **RAGFlow-Plus** |
| 高精度文档解析 | RAGFlow-Plus + MinerU |

## 三、企业级权限体系设计

### 3.1 多级权限架构

```
┌─────────────────────────────────────────────────────────────┐
│                    系统超级管理员                             │
│              （全局配置、库创建审批、系统监控）                  │
└─────────────────────────────────────────────────────────────┘
                              │
            ┌─────────────────┼─────────────────┐
            ▼                 ▼                 ▼
    ┌───────────────┐ ┌───────────────┐ ┌───────────────┐
    │   库超级管理员  │ │   库超级管理员  │ │   库超级管理员  │
    │  （知识库A）    │ │  （知识库B）    │ │  （知识库C）    │
    └───────────────┘ └───────────────┘ └───────────────┘
            │                 │                 │
      ┌─────┴─────┐     ┌─────┴─────┐     ┌─────┴─────┐
      ▼           ▼     ▼           ▼     ▼           ▼
   编辑者      查看者  编辑者      查看者  编辑者      查看者
```

### 3.2 角色权限矩阵

| 角色 | 英文名 | 权限说明 | 适用人群 |
|------|--------|----------|----------|
| 系统管理员 | SuperAdmin | 全局配置、库创建审批、用户管理 | IT管理员 |
| 库超级管理员 | Owner | 库内全权限、用户授权、配置管理 | 部门负责人 |
| 库编辑者 | Editor | 上传/编辑/删除文档、无用户管理 | 业务骨干 |
| 库查看者 | Viewer | 查看文档、批注（可配置下载权） | 协作人员 |
| 库访客 | Guest | 仅公开目录查看、无下载/转发 | 临时人员 |

### 3.3 数据隔离策略

**库级隔离**：
- 每个知识库独立存储
- 跨库检索需显式授权

**目录级隔离**：
- 按目录分级授权
- 权限可向下收紧，不可向上突破

**文档级隔离**：
- 单文档精准授权
- 创建者可自主分享权限

## 四、RAG 行级权限设计

### 4.1 行级权限原理

在向量检索时，通过 **元数据过滤（Metadata Filtering）** 实现行级权限控制：

```python
# 伪代码示例
def search_with_permission(query, user):
    # 获取用户可见的数据范围
    user_departments = user.departments  # ["研发部", "产品部"]
    user_level = user.access_level       # "confidential"
    
    # 构建过滤条件
    filters = {
        "department": {"$in": user_departments},
        "security_level": {"$lte": user_level}
    }
    
    # 执行带过滤的向量检索
    results = vector_store.search(
        query=query,
        filters=filters,
        top_k=10
    )
    
    return results
```

### 4.2 元数据设计

为每个文档切片添加权限元数据：

```json
{
  "chunk_id": "doc_001_chunk_003",
  "content": "产品X的技术参数...",
  "embedding": [0.123, 0.456, ...],
  "metadata": {
    "doc_id": "doc_001",
    "department": "研发部",
    "security_level": "internal",      // public/internal/confidential
    "project": "项目A",
    "owner": "user_123",
    "created_at": "2026-04-24",
    "tags": ["技术文档", "产品规格"]
  }
}
```

### 4.3 权限过滤流程

```
用户提问 → 解析用户权限 → 构建过滤条件 → 向量检索 → 返回授权结果
                │
                ├── 部门权限：用户所属部门
                ├── 项目权限：用户参与项目
                ├── 密级权限：用户访问级别
                └── 所有者权限：用户创建的文档
```

## 五、企业级知识库增强策略

### 5.1 检索增强策略

| 策略 | 说明 | 效果提升 |
|------|------|----------|
| **混合检索** | BM25 + 向量检索融合 | 召回率+25% |
| **二次排序** | Cross-Encoder 重排序 | 精确率+40% |
| **多轮改写** | 结合对话历史补全查询 | 上下文理解+30% |
| **标签过滤** | 按文档标签预筛选 | 召回速度+50% |

### 5.2 知识图谱融合（GraphRAG）

```
┌─────────────────────────────────────────────────────────────┐
│                     GraphRAG 增强流程                        │
└─────────────────────────────────────────────────────────────┘

用户提问："产品A的售后服务流程是什么？"
        │
        ▼
┌───────────────────┐
│ 实体识别          │ → 识别出"产品A"、"售后服务"、"流程"
└───────────────────┘
        │
        ▼
┌───────────────────┐
│ 图谱关联查询       │ → 产品A -[属于]-> 产品线B
│                   │ → 产品线B -[售后政策]-> 通用条款
└───────────────────┘
        │
        ▼
┌───────────────────┐
│ 扩展检索          │ → 同时检索产品A和产品线B的售后文档
└───────────────────┘
        │
        ▼
┌───────────────────┐
│ 事实校验          │ → 图谱规则验证答案准确性
└───────────────────┘
```

### 5.3 幻觉防御机制

| 层级 | 防御措施 | 说明 |
|------|----------|------|
| **检索层** | 相似度阈值过滤 | 低相关性内容不召回 |
| **生成层** | 知识图谱校验 | 关键事实与图谱对比 |
| **输出层** | 引用追溯 | 答案必须标注来源 |

### 5.4 持续优化闭环

```
用户反馈 → 问题标注 → 数据收集 → 模型优化 → 效果评估 → 上线部署
    ↑                                                        │
    └────────────────────────────────────────────────────────┘
```
