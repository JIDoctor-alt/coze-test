## 一、项目概述

### 1.1 项目背景

客户已通过知识库汇报，需要交付可商用的企业级知识库平台。技术选型为 RAGFlow 开源版本进行二开，参考 RAGFlow-Plus 的企业级改造方案。

### 1.2 项目目标

- 构建企业级知识库管理系统
- 实现多租户、多权限体系
- 支持高精度文档解析（MinerU）
- 提供 RAG 智能问答能力
- 支持行级权限控制

### 1.3 核心需求

| 需求 | 说明 | 优先级 |
|------|------|--------|
| 部署方案 | 阿里云服务 / 自建 RAGFlow | P0 |
| 系统设计 | 企业级架构 + RAG 集成 | P0 |
| 权限体系 | 用户权限 + 数据隔离 | P0 |
| 行级权限 | RAG 检索的权限过滤 | P0 |
| 增强策略 | GraphRAG、混合检索等 | P1 |

## 三、权限体系设计

### 3.1 权限模型

采用 **RBAC + ABAC 混合模型**：

```
RBAC（基于角色）
    └── 用户 → 角色 → 权限
    
ABAC（基于属性）
    └── 用户属性 + 资源属性 + 环境属性 → 访问决策
```

### 3.2 多级权限架构

```
系统级
  │
  ├── 租户（Tenant）
  │     │
  │     ├── 知识库（Knowledge Base）
  │     │     │
  │     │     ├── 目录（Folder）
  │     │     │     │
  │     │     │     └── 文档（Document）
  │     │     │
  │     │     └── 文档（Document）
  │     │
  │     └── 用户组（User Group）
  │           │
  │           └── 用户（User）
  │
  └── 系统配置
```

### 3.3 角色定义

| 角色 | 系统权限 | 库权限 | 文档权限 |
|------|----------|--------|----------|
| 超级管理员 | 全部 | 全部 | 全部 |
| 租户管理员 | 租户配置 | 创建/管理库 | 全部 |
| 库管理员 | - | 库配置/用户授权 | 全部 |
| 编辑者 | - | 上传/编辑 | 创建/编辑 |
| 查看者 | - | 查看 | 查看 |
| 访客 | - | 公开内容 | 公开内容 |

### 3.4 数据隔离方案

**1. 租户隔离**
```sql
-- 所有表增加 tenant_id 字段
CREATE TABLE knowledge_base (
    id UUID PRIMARY KEY,
    tenant_id UUID NOT NULL,
    name VARCHAR(255),
    -- 强制租户隔离
    CONSTRAINT tenant_fk FOREIGN KEY (tenant_id) REFERENCES tenant(id)
);

-- 查询时自动添加租户过滤
SELECT * FROM knowledge_base WHERE tenant_id = :current_tenant_id;
```

**2. 库级隔离**
```sql
-- 文档表关联知识库
CREATE TABLE document (
    id UUID PRIMARY KEY,
    kb_id UUID NOT NULL,
    -- 用户权限通过 kb_id 关联
);
```

**3. 行级权限实现**
```python
class DocumentService:
    def get_documents(self, user, kb_id, filters=None):
        # 1. 检查用户是否有库访问权限
        if not self.check_kb_access(user, kb_id):
            raise PermissionDenied()
        
        # 2. 获取用户的行级过滤条件
        row_filters = self.get_row_filters(user)
        
        # 3. 合并过滤条件
        query = Document.query.filter_by(kb_id=kb_id)
        query = query.filter(row_filters)  # 行级权限
        
        return query.all()
    
    def get_row_filters(self, user):
        """根据用户属性构建行级过滤条件"""
        conditions = []
        
        # 部门过滤
        if user.department:
            conditions.append(
                Document.department.in_(user.accessible_departments)
            )
        
        # 项目过滤
        if user.projects:
            conditions.append(
                Document.project.in_(user.projects)
            )
        
        # 密级过滤
        conditions.append(
            Document.security_level <= user.security_level
        )
        
        return and_(*conditions)
```

### 4.3 行级权限在 RAG 中的实现

```python
class RAGService:
    def search_with_permission(
        self, 
        query: str, 
        user: User,
        kb_ids: List[str],
        top_k: int = 10
    ) -> List[SearchResult]:
        
        # 1. 获取用户权限属性
        user_attrs = self.get_user_attributes(user)
        """
        user_attrs = {
            "departments": ["研发部", "产品部"],
            "projects": ["项目A", "项目B"],
            "security_level": "internal",  # public < internal < confidential
            "roles": ["editor", "viewer"]
        }
        """
        
        # 2. 构建元数据过滤条件
        filters = self.build_permission_filters(user_attrs)
        """
        filters = {
            "bool": {
                "must": [
                    {"terms": {"department": ["研发部", "产品部"]}},
                    {"terms": {"project": ["项目A", "项目B"]}},
                    {"term": {"security_level": "public"}},
                ]
            }
        }
        """
        
        # 3. 向量检索（带权限过滤）
        vector_results = self.vector_search(
            query=query,
            kb_ids=kb_ids,
            filters=filters,  # 权限过滤
            top_k=top_k * 2   # 多召回一些，供重排序
        )
        
        # 4. BM25 检索（带权限过滤）
        keyword_results = self.keyword_search(
            query=query,
            kb_ids=kb_ids,
            filters=filters,  # 权限过滤
            top_k=top_k * 2
        )
        
        # 5. 融合重排序
        merged = self.merge_and_rerank(
            vector_results, 
            keyword_results,
            top_k
        )
        
        return merged
```

## 六、数据库设计

### 6.1 核心表结构

```sql
-- 租户表
CREATE TABLE tenant (
    id UUID PRIMARY KEY,
    name VARCHAR(255) NOT NULL,
    status VARCHAR(20) DEFAULT 'active',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 用户表
CREATE TABLE user (
    id UUID PRIMARY KEY,
    tenant_id UUID NOT NULL,
    username VARCHAR(100) NOT NULL,
    email VARCHAR(255),
    department VARCHAR(100),
    security_level VARCHAR(20) DEFAULT 'public',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (tenant_id) REFERENCES tenant(id)
);

-- 知识库表
CREATE TABLE knowledge_base (
    id UUID PRIMARY KEY,
    tenant_id UUID NOT NULL,
    name VARCHAR(255) NOT NULL,
    description TEXT,
    owner_id UUID,
    status VARCHAR(20) DEFAULT 'active',
    config JSONB,  -- 向量模型、切片策略等配置
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (tenant_id) REFERENCES tenant(id),
    FOREIGN KEY (owner_id) REFERENCES user(id)
);

-- 文档表
CREATE TABLE document (
    id UUID PRIMARY KEY,
    kb_id UUID NOT NULL,
    file_name VARCHAR(500),
    file_path VARCHAR(1000),
    file_type VARCHAR(50),
    file_size BIGINT,
    status VARCHAR(20),  -- pending/parsing/completed/failed
    department VARCHAR(100),
    project VARCHAR(100),
    security_level VARCHAR(20) DEFAULT 'public',
    tags JSONB,
    created_by UUID,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (kb_id) REFERENCES knowledge_base(id),
    FOREIGN KEY (created_by) REFERENCES user(id)
);

-- 文档切片表
CREATE TABLE document_chunk (
    id UUID PRIMARY KEY,
    doc_id UUID NOT NULL,
    kb_id UUID NOT NULL,
    chunk_index INTEGER,
    content TEXT NOT NULL,
    embedding VECTOR(1024),  -- 向量字段
    metadata JSONB,  -- 权限元数据
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (doc_id) REFERENCES document(id),
    FOREIGN KEY (kb_id) REFERENCES knowledge_base(id)
);

-- 权限表
CREATE TABLE permission (
    id UUID PRIMARY KEY,
    kb_id UUID,
    user_id UUID,
    role VARCHAR(50),  -- owner/editor/viewer/guest
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (kb_id) REFERENCES knowledge_base(id),
    FOREIGN KEY (user_id) REFERENCES user(id)
);
```

## 八、开发计划

### 8.1 阶段划分

| 阶段 | 内容 | 工期 | 交付物 |
|------|------|------|--------|
| **Phase 1** | 基础架构搭建 | 2周 | 部署文档、基础API |
| **Phase 2** | 权限体系实现 | 2周 | 多级权限、数据隔离 |
| **Phase 3** | RAG 核心功能 | 3周 | 文档解析、智能问答 |
| **Phase 4** | 增强功能 | 2周 | GraphRAG、文档撰写 |
| **Phase 5** | 测试与优化 | 2周 | 测试报告、性能优化 |

### 8.2 技术风险

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| MinerU GPU 资源不足 | 解析速度慢 | 使用 API 服务或云 GPU |
| 向量库性能瓶颈 | 检索延迟高 | 集群部署、索引优化 |
| LLM 幻觉问题 | 答案不准确 | 引用追溯、知识图谱校验 |
