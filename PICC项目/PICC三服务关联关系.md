## 一、先认识三个"服务员"

想象一下，你去一家大型餐厅吃饭。这家餐厅有三个超级重要的服务员：

| 服务员 | 花名（项目名） | 工牌号（端口） | 职责 | 手下小弟数量 |
|--------|---------------|---------------|------|-------------|
| **前台接待** | picc-mzmtb-gateway | 9001 | 接待客人、传菜单、端菜 | 809个Java文件 |
| **权限管家** | picc-mzmtb-user | 9092 | 查身份证、验证身份 | 136个Java文件 |
| **业务达人** | picc-mzmtb-server | 9091 | 后厨做菜，慢特病业务全流程 | 2647个Java文件 |

### 前台接待（picc-mzmtb-gateway）—— 新加入！

```
🎯 核心能力：HTTP请求转发、VO层转换、小程序接口、医保系统对接
🔧 技术栈：Spring Boot + HttpClient + CXF WebService + Redis + Apollo
📊 数据规模：361个VO类（点餐单）、809个Java文件
🎒 特殊角色：
   ✅ 361个VO（点餐单）—— 只展示客人需要看的信息，隐藏后厨细节
   ✅ 小程序接口 —— 接待手机来的客人
   ✅ CXF WebService —— 和医保局系统对接
```

### 权限管家（picc-mzmtb-user）

```
🎯 核心能力：用户登录、权限管理、机构管理、角色管理、菜单管理
🔧 技术栈：Spring Boot + MyBatis + GaussDB + Redis + Apollo
📊 数据规模：17张业务表、136个Java文件
🎒 小弟数量：规模较小（只管身份）
```

### 业务达人（picc-mzmtb-server）

```
🎯 核心能力：慢特病申报、审批、发卡、处方、药店、费用管理
🔧 技术栈：Spring Boot + MyBatis + Activiti6 + GaussDB + Redis + Apollo
📊 数据规模：213个Mapper、260+个PO实体、894个API接口、2647个Java文件
🎒 小弟数量：是权限服务的19倍！（业务服务规模巨大）
```

## 二、架构关系：三个服务是怎么住在一起的？

### 2.1 部署关系（K8s中各过各的）

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                          K8s 集群（相当于一个大型写字楼）                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐            │
│   │   Pod: gateway  │  │   Pod: user     │  │   Pod: server   │            │
│   │ ┌─────────────┐ │  │ ┌─────────────┐ │  │ ┌─────────────┐ │            │
│   │ │   前台接待   │ │  │ │   权限管家   │ │  │ │   业务达人   │ │            │
│   │ │   9001端口  │ │  │ │   9092端口  │ │  │ │   9091端口  │ │            │
│   │ └─────────────┘ │  │ └─────────────┘ │  │ └─────────────┘ │            │
│   └────────┬────────┘  └────────┬────────┘  └────────┬────────┘            │

             │                   │                   │
             │  ← 各自独立部署，互不依赖Pod重启  │
```

**特点**：
- ✅ **独立Pod**：三个服务各自住在自己的"单间"里，一个挂了不影响另外两个
- ✅ **独立Deployment**：各自的K8s部署配置分开管理
- ✅ **独立副本数**：业务服务可能跑3个副本，前台和权限服务可能跑2个副本

**前端调用方式**：
- 🎯 **统一入口**：所有前端请求先到前台服务9001
- 🎯 **前台路由**：前台服务根据URL路径，把请求**HTTP转发**到后端
- 🎯 **隐藏细节**：前端不需要知道权限服务和业务服务的具体地址
- 🎯 **VO转换**：前台服务把后端返回的"大份数据"变成"小份点餐单"返回前端

### 3.2 前台服务的361个VO类（点餐单）

```
想象你去餐厅点菜：

❌ 服务员不会把后厨的所有食材都给你看
✅ 服务员只会给你一张"点餐单"，上面写着：
   - 菜品名称（VO字段）
   - 价格（展示数据）
   - 数量（可控范围）

同理，前台服务的361个VO类：
- 把业务服务的260+个PO实体（原材料）转换成361个VO（点餐单）
- 前端只看到"需要展示什么"，不需要知道"怎么存储"
- 隐藏内部数据结构和业务逻辑
```

### 4.4 HTTP转发机制（HttpForwardUtil）

**前台服务怎么"打电话"给业务服务？**

```java
// 前台服务中的HTTP转发工具（类似对讲机）
@Component
public class HttpForwardUtil {
    
    // 转发到业务服务
    public String forwardToServer(String url, Map<String, Object> params) {
        // 1. 把请求参数转成JSON
        String jsonBody = JSON.toJSONString(params);
        
        // 2. 加上Token（从请求头获取）
        String token = getTokenFromHeader();
        
        // 3. 发送HTTP POST请求到业务服务9091
        HttpResponse response = HttpUtil.createPost("http://server:9091" + url)
            .header("Authorization", token)
            .body(jsonBody)
            .execute();
        
        // 4. 返回业务服务的响应
        return response.body();
    }
    
    // 转发到权限服务（验证Token）
    public UserInfo forwardToUserForToken(String token) {
        // 发送HTTP请求到权限服务9092验证Token
        HttpResponse response = HttpUtil.createGet("http://user:9092/privilege/auth/checkToken")
            .query("token", token)
            .execute();
        
        return JSON.parseObject(response.body(), UserInfo.class);
    }
}
```

**转发示例**：

```java
@Service
public class GatewayService {
    
    @Autowired
    private HttpForwardUtil httpForwardUtil;
    
    // 前台接收"查询申报列表"请求
    public DeclareListVO queryDeclareList(String token, QueryForm form) {
        // 1. 先问权限服务：这个Token有效吗？
        UserInfo user = httpForwardUtil.forwardToUserForToken(token);
        
        if (user == null) {
            throw new BizException("请先登录");
        }
        
        // 2. 转发给业务服务：帮我查申报列表
        String serverResponse = httpForwardUtil.forwardToServer("/api/mb/declare/list", form);
        
        // 3. VO转换：把业务服务的大数据变成前台的小数据
        DeclareListVO vo = convertToVO(serverResponse);
        
        return vo;
    }
}
```

## 八、常见问题FAQ

### Q1：为什么需要前台服务？
> **答**：三个原因
> 1. **统一入口**：前端不需要记住多个服务地址，只记前台9001
> 2. **VO转换**：后端返回的"原材料"太多，前台帮你筛选成"点餐单"
> 3. **协议适配**：前台能对接小程序、医保局WebService等多种客户端

### Q2：前台服务挂了，业务还能用吗？
> **答**：**不能用！**
> - 前台是统一入口，前端所有请求都先到前台
> - 前台挂了 = 所有请求都进不来
> - 但业务服务和权限服务之间可以正常调用

### Q3：三个服务用同一个Redis，会有冲突吗？
> **答**：**不会**
> - Redis Key有前缀区分：`token:*`、`user:*`、`cache:*`
> - 不同服务的Key不会互相覆盖
> - 但如果Redis挂了，**三个服务都会受影响**

### Q4：前台服务为什么不直接调用数据库？
> **答**：**因为前台是"服务员"，不是"厨师"**
> - 服务员负责：接待客人、传话、端菜
> - 厨师负责：做菜、存储食材
> - 前台只做HTTP转发和VO转换，不碰数据库

### Q5：权限服务挂了会怎样？
> **答**：**部分功能受影响**
> - ✅ 业务服务的**已有请求**：可能正常（如果Token还在Redis中）
> - ❌ **新登录**：不行（登录接口在权限服务）
> - ❌ **Token验证**：不行（前台无法验证Token）
> - ❌ **权限校验**：不行（业务服务Feign会超时）

### Q6：361个VO类是什么意思？
> **答**：**361张"点餐单"**
> - 后端业务服务有260+个PO实体（原材料/食材）
> - 前台服务有361个VO类（点餐单/成品菜）
> - VO = View Object，视图对象，只展示前端需要的字段
