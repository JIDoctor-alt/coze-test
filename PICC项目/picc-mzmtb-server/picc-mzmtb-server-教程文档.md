## 📋 各文档核心发现摘要

### 1️⃣ 项目全貌（102行）
- **核心结论**：业务服务是权限服务的19倍规模，mb模块(833文件)是绝对主力
- **架构亮点**：采用"中央厨房+本地厨师"模式，mtb-base(674文件)做基础，12个地市模块做差异化
- **服务关系**：业务服务(9091)通过Feign调用权限服务(9092)的用户/角色/菜单接口

### 2️⃣ 架构解析（782行）
- **技术栈**：Spring Boot 4.2.6.0 + MyBatis + Activiti 6.0.0 + GaussDB + Redis(Redisson 3.13.4) + Apollo
- **信创适配**：BES应用服务器 + GaussDB数据库，通过activiti-engine-picc定制版解决兼容
- **文档处理**：POI 4.1.2 + PDFBox 2.0.21 + JodConverter 4.3.0 + iTextPDF 5.5.10
- **核心问题**：使用自定义Interceptor而非Spring Security做权限控制

### 3️⃣ API接口全景（1016行）
| 业务域 | 接口数 | 占比 | 核心API |
|--------|--------|------|---------|
| 申报管理 | 435 | 48.7% | DocDeclareApi、MbDeclareApi、XcxIndexApi |
| 系统管理 | 155 | 17.3% | 菜单、角色、机构管理 |
| 处方管理 | 87 | 9.7% | MbPrescriptionApi、PrescriptionPoolApi |
| 慢病卡管理 | 52 | 5.8% | VipCardApi、MbCardStatusApi |
| 审核管理 | 36 | 4.0% | FirstTrialApi、ReviewApi |

### 4️⃣ 申报流程解析（1164行）
```
开始申报 → 材料提交(线上/线下/批量) → 初审 → [体检] → 专家审核 → 发卡 → 待遇享受
```
- **状态机**：10个状态(审核中/初审通过/初审驳回/审核通过/驳回/部分通过/复审通过/补充资料/无效)
- **体检机制**：最多重试3次，未到可放弃
- **发卡类型**：新发卡/无变化/信息变更，支持邮寄或自取

### 5️⃣ 数据模型解析（1159行）
- **核心8表**：VIP_MBDECLARE_INFO(申报)、MB_FIRSTTRIAL(初审)、MB_REVIEW(复审)、MB_PERSON(人员)、MB_DISEASE(疾病)、VIP_PRESCRIPTION(处方)、VIP_CARD(卡)、GHI_*(医保)
- **命名规律**：VIP_=会员、MB_=慢特病、GHI_=医保社保
- **敏感字段加密**：name/idcard/mobile使用EncryptionUtil加密存储

### 6️⃣ 安全与代码质量审计（1348行）
| 等级 | 数量 | 代表性问题 |
|------|------|-----------|
| P0紧急 | 8个 | 未使用安全框架(107接口)、硬编码密码、MD5无盐值、敏感字段明文传输 |
| P1高危 | 15个 | 缺失CSRF防护、Redis无密码、Nacos未授权、文件上传无限制 |
| P2中危 | 10个 | SQL日志泄露、弱注释规范、日志脱敏不完整 |

### 7️⃣ 地市差异化机制（771行）
- **路由机制**：通过`flag`参数和Service BeanName选择实现类
- **13地市**：西安/咸阳/延安/榆林/商洛/宝鸡/晋城/九江/达州/眉县/德州/榆林(独立)/海南
- **差异类型**：申报规则、审批流程、Excel模板、医保对接方式
- **接入7步法**：继承base类→注册Bean→配置flag→实现差异化方法→测试→上线

### 8️⃣ 处方与药店管理解析（1124行）
- **处方管理**：87接口，支持图片上传、OCR识别(延安)、药品录入、驳回修改
- **药店管理**：38接口，缴费管理、订单处理、药品目录维护
- **地市差异**：榆林/宝鸡/延安/商洛各有独立处方表和Service实现
- **费用管理**：结算对账、退费操作、医保对接
