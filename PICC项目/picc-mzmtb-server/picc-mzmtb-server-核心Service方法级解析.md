## 术语对照表（小白的冰箱贴）

| 术语 | 人话解释 |
|------|----------|
| `@Transactional` | 银行转账，要么全成功要么全回滚（要么全部数据库操作都完成，要么一步失败全部撤销） |
| `@Cacheable` | 冰箱贴，常用数据贴冰箱不用每次翻（把查询结果缓存起来，下次直接用） |
| `Mapper` | 菜谱，告诉数据库按什么方子找数据 |
| `Feign调用` | 打电话给另一个部门帮忙干活 |
| `ServiceImpl` | 业务逻辑的"厨师长"，负责炒菜（处理业务） |
| `Dao` | 数据库操作的"仓库管理员"，负责存取数据 |
| `Activiti` | 工作流引擎，就像流水线加工，任务沿着固定流程走 |

## 1. VIPChronicPayServiceImpl

**慢病支付服务 - 医保卡充值、消费、退款的"钱袋子管家"**

### 核心依赖
```
- VipInfoService: 会员信息服务
- VipAccountService: 会员账户服务
- VipMoneyflowService: 资金流水服务
- VipPayService: 支付服务
- VipDrugstoreOrderService: 药店订单服务
- MedicalInsuranceService: 医保接口服务（打电话给医保系统）
```

### 方法解析

---

#### 1.1 `VIPAuthChronic()` - 慢病身份认证

**📌 一句话解释：** 验证用户是否是合法的慢病患者，就像刷卡进门

**参数说明：**
| 参数 | 类型 | 说明 |
|------|------|------|
| request | XMLTable<VIPAuthChronicRequestVo> | 包含卡号、密码、身份证号等身份信息 |

**返回值：** `ApiResponse<VIPAuthChronicResponseVo>` - 认证是否成功

**核心逻辑拆解：**
```
1️⃣ 检查业务ID是否为"CH002"（身份认证的专属编号）
   ↓
2️⃣ 根据卡号查账户 → 找不到再用身份证+姓名查
   ↓
3️⃣ 如果找到多个账户（一人多卡），遍历检查：
   - 跳过冻结的卡
   - 检查本月是否已购药
   ↓
4️⃣ 验证密码（MD5加密后比对）
   ↓
5️⃣ 检查是否有未过期的复审资格
   ↓
6️⃣ 返回认证结果
```

**异常处理：**
- 密码错误 → "输入密码错误，请重新输入"
- 卡已冻结 → "慢病卡已冻结"
- 复审过期 → "复审已过期，请及时联系慢病窗口进行复审"
- 本月已购药 → "本月已经完成购药"

**💡 小白易懵点：**
> 一个身份证号可能对应多张慢病卡（职工卡、居民卡等），代码会遍历所有卡找一张能用的。就像一个人可能有多个银行账户，系统会找余额最多的那个。

---

#### 1.2 `VIPQueryChronicBalance()` - 查询账户余额

**📌 一句话解释：** 查询慢病账户里还剩多少钱

**参数说明：**
| 参数 | 类型 | 说明 |
|------|------|------|
| request | XMLTable<VIPQueryChronicBalanceRequestVo> | 包含卡号 |

**核心逻辑：**
```
1️⃣ 业务ID检查：必须是"CH001"
2️⃣ 根据卡号查账户列表
3️⃣ 统计每个账户：
   - 当月起始金额
   - 当月已消费
   - 当月剩余额度
4️⃣ 计算汇总返回
```

---

#### 1.3 `VIPChronicPay()` - 慢病支付（核心交易）

**📌 一句话解释：** 患者买药后走医保报销，系统记账扣钱

**@Transactional 注解解读：** 
> 就像ATM转账：查询余额→扣款→记录流水，要么全成功，要么卡纸了全部回退

**核心逻辑：**
```
1️⃣ 业务ID检查：必须是"CH003"

2️⃣ 查找会员和账户
   - 用卡号或订单号找账户
   - 验证账户状态是否正常

3️⃣ 扣款处理
   - 计算报销金额（总费用 - 自付比例）
   - 更新账户余额
   - 记录资金流水

4️⃣ 药品库存更新
   - 药店药品库存 -1

5️⃣ 处方信息更新
   - 购买次数 +1
```

**⚠️ 异常处理：**
- 当天之前不能报销
- 只能当月退款
- 库存不足拒绝交易

---

#### 1.4 `VIPChronicRefund()` - 慢病退款

**📌 一句话解释：** 买错了药要退钱，把报销的钱退回去

**核心逻辑：**
```
1️⃣ 业务ID检查：必须是"CH004"

2️⃣ 参数校验
   - 订单号、卡号必填
   - 退款金额必填

3️⃣ 资金退款（银行转账回滚机制）
   - 恢复账户月起始金额
   - 记录退款流水

4️⃣ 库存恢复
   - 药品库存 +1

5️⃣ 处方购买次数 -1

6️⃣ 更新订单状态为"已退款"
```

---

#### 1.5 `VIPChronicPayCallback()` - 支付回调

**📌 一句话解释：** 第三方支付平台通知"钱已到账"，系统更新订单状态

**核心逻辑：**
```
1️⃣ 防重复处理（接口幂等性）
   - 用订单号查是否已处理过
   - 已处理直接返回

2️⃣ 更新订单信息
   - 账户金额
   - 总金额
   - 预付款金额
   - 回传标识
```

---

#### 1.6 `VIPUpdatePassword()` - 修改密码

**📌 一句话解释：** 修改慢病卡消费密码

**核心逻辑：**
```
1️⃣ 业务ID检查：必须是"CH006"

2️⃣ 查找会员信息（用身份证）

3️⃣ 密码校验
   - 新密码不能等于旧密码
   - 新密码不能是身份证尾号（太容易被猜到）

4️⃣ 更新所有关联账户的密码
```

**💡 小白易懵点：**
> 一个会员可能有多张慢病卡（职工卡/居民卡），修改密码会同时修改所有卡，避免"这张卡能用那张不能用"的尴尬。

## 2. MtbDeclareExpertAssignServiceImpl

**专家分配服务 - 给慢病鉴定任务分配"裁判"的调度员**

### 核心依赖
```
- TaskService: Activiti工作流任务服务
- TVipMbdeclareInfoService: 申报信息服务
- TVipReviewInfoService: 复审信息服务
- TVipMbdeclareApprovalService: 审批信息服务
```

### 方法解析

---

#### 2.1 `query()` - 查询体检完成的申报列表

**📌 一句话解释：** 找出已完成体检、等待专家分配的任务

**参数说明：**
| 参数 | 类型 | 说明 |
|------|------|------|
| selectVipMbDeclareInfoVo | QueryMbInfoVo | 查询条件（分页、申报状态等） |

**返回值：** 分页的申报信息列表

**核心逻辑：**
```
1️⃣ 根据数据范围标识(flag)确定查询权限
2️⃣ 分页参数处理
3️⃣ 判断是否启用工作流
   - 不启用：直接查数据库
   - 启用：查工作流+关联审批信息
4️⃣ 补充审批人信息
```

---

#### 2.2 `updateInfo()` - 专家手动撤回

**📌 一句话解释：** 专家发现自己分错了任务，把任务退回去

**核心逻辑：**
```
1️⃣ 处理申报主表撤回
   - 清空二审专家ID
   - 清空二审时间

2️⃣ 工作流任务解绑
   - taskService.unclaim() 释放任务

3️⃣ 处理复审专家撤回
   - 类似逻辑，清空复审专家
```

**💡 小白易懵点：**
> "unclaim"不是删除任务，是"放弃认领"，就像领了任务单后发现不对，放回任务箱让别人领。

---

#### 2.3 `getVipMbdeclareFileTypesByDeclareId()` - 获取申报附件

**📌 一句话解释：** 根据申报ID查找上传的证明材料

**核心逻辑：**
```
1️⃣ 支持文件类型过滤
2️⃣ 处理宝鸡病种拆分逻辑（特殊业务）
3️⃣ 查询文件列表并返回
```

---

#### 2.4 `physicalAssign()` - 体检机构分配

**📌 一句话解释：** 把申报人分配到指定医院做体检

**核心逻辑：**
```
1️⃣ 批量处理申报ID列表

2️⃣ 更新申报主表状态
   - 体检状态 = 体检中
   - 机构状态 = 进行中
   - 预约标识 = 已预约

3️⃣ 创建/更新体检信息
   - 记录体检机构编码和名称
   - 记录操作人

4️⃣ 工作流任务认领
   - 分配给指定医院账号
```

---

#### 2.5 `updateExpert()` - 手动分配专家

**📌 一句话解释：** 人工给申报分配审核专家

**核心逻辑：**
```
1️⃣ 参数校验
   - 不能重复分配

2️⃣ 更新审批记录
   - 记录专家ID、姓名、账号

3️⃣ 创建专家鉴定记录
   - 记录病种和专家对应关系

4️⃣ 更新申报信息（二审时间）

5️⃣ 发送短信通知专家
   - "您有一条新的慢病审核信息需要处理"
```

---

#### 2.6 `queryMbExpertList()` - 查询专家列表

**📌 一句话解释：** 根据病种查找符合条件的审核专家

**核心逻辑：**
```
1️⃣ 分页处理
2️⃣ 病种编码排序拼接（"M001,M002,M003"）
3️⃣ 获取专家角色ID（慢病专家角色）
4️⃣ 按条件查询：病种匹配 + 数据权限
```

---

#### 2.7 `meansBackExpert()` - 专家退回申报

**📌 一句话解释：** 专家审核后认为材料不全，退回让重新提交

**@Transactional 解读：**
> 三步操作必须同时成功：删文件→改状态→推进工作流

**核心逻辑：**
```
1️⃣ 删除关联文件
2️⃣ 更新申报状态 = 体检中（退回到体检阶段）
3️⃣ 重置申请状态 = 待审核
4️⃣ 工作流节点流转（secondApproveFlag = 2）
```

## 4. VipReviewServiceImpl

**复审服务 - 慢病卡到期"换证"的审核流水线**

### 核心依赖
```
- VipReviewInfoDao: 复审信息数据库操作
- ActivitiDao: 工作流数据库操作
- RuntimeService: 工作流运行服务
- TaskService: 工作流任务服务
```

### 核心概念

| 概念 | 说明 |
|------|------|
| 复审状态 | 0-未到复审期, 1-复审预警, 2-复审中, 3-复审拒绝, 4-复审不通过, 5-复审通过, 6-复审不通过 |
| 体检状态 | 0-体检中, 1-体检完成, 4-未到体检, 5-放弃体检 |
| 结案状态 | 1-进行中, 2-已结案 |

### 方法解析

---

#### 4.1 `queryReview()` - 查询复审列表

**📌 一句话解释：** 查找需要复审或正在复审的慢病记录

**核心逻辑：**
```
1️⃣ 分页处理
2️⃣ 根据地区标识(flag)判断业务逻辑
   - 宝鸡：走特殊查询
   - 其他：走通用查询
3️⃣ 判断是否启用工作流
4️⃣ 执行查询并分页
```

---

#### 4.2 `exportReviewList()` - 导出复审清单

**📌 一句话解释：** 把复审数据导出成Excel表格

**核心逻辑：**
```
1️⃣ 先查询数据
2️⃣ 打开导出模板
3️⃣ 按行填充数据
   - 过滤条件：连续3年享受待遇且满65周岁的不导出
   - 各种状态的数据过滤
4️⃣ 添加水印
5️⃣ 返回Base64编码
```

---

#### 4.3 `batchReview()` - 批量复审

**📌 一句话解释：** 一键给多个到期患者办理复审

**@Transactional 解读：**
> 更新数据库→创建工作流→完成工作流节点，必须一起成功

**核心逻辑：**
```
1️⃣ 遍历每条待复审数据

2️⃣ 状态校验（不符合条件跳过并记录原因）
   - 结案数据不能复审
   - 已过期3个月不能复审
   - 未到复审期不能复审

3️⃣ 允许复审则：
   - 清除二审专家
   - 设置复审状态 = 复审中
   - 设置体检状态 = 未分配
   - 设置结案状态 = 进行中
   - 保存到数据库

4️⃣ 工作流处理
   - 新任务：启动新工作流
   - 进行中任务：推进工作流节点
```

**💡 小白易懵点：**
> 批量处理允许"部分成功"，比如10条数据成功8条失败2条，返回"有数据已完成复审[2条]"。

---

#### 4.4 `refuseReview()` - 拒绝复审

**📌 一句话解释：** 审核人员认为不符合条件，拒绝复审

**核心逻辑：**
```
1️⃣ 更新复审状态 = 3（复审拒绝）
2️⃣ 设置复审日期
3️⃣ 设置结案状态 = 已结案
4️⃣ 推进工作流节点
```

## 5. VipJkdaServiceImpl

**健康档案服务 - 对接卫健委健康档案系统的"翻译官"**

### 核心依赖
```
- BaseCallService: 基础调用服务（打电话给外部系统）
- VipMbdeclareInfoDao: 申报信息数据库
- VipReviewInfoDao: 复审信息数据库
- TaskService: 工作流任务服务
```

### 方法解析

---

#### 5.1 `QueryAndUpPhyReport()` - 查询并上传体检报告

**📌 一句话解释：** 从健康档案系统拉取体检报告，同步到申报系统

**核心逻辑：**
```
1️⃣ 查询申报信息（根据申报ID）
   - 找不到再查复审信息

2️⃣ 查询体检信息（根据体检ID）

3️⃣ 判断医院是否对接
   - 有fileorgcode = 已对接，可以自动拉取
   - 无 = 未对接，只能手动上传

4️⃣ 调用健康档案接口
   - 发送：姓名、性别、生日、证件号、体检机构
   - 接收：体检报告PDF或数据

5️⃣ 解析结果
   - 有报告：更新体检状态=完成，保存文件
   - 无报告：返回提示"暂无体检报告"

6️⃣ 推进工作流节点
```

---

#### 5.2 `QueryAndUpPhyReportRe()` - 复审体检报告查询上传

**📌 一句话解释：** 复审流程中的体检报告同步

**与上面方法的区别：**
- 上面查首次申报的体检报告
- 这个查复审的体检报告

#### 7.1 `queryFiling()` - 查询备案列表

**核心逻辑：**

   - 失败：记录错误原因，状态=备案失败

4️⃣ 更新本地备案明细表
   - 备案时间
   - 备案状态
   - 医保返回的流水号

@PostConstruct
public void initVipPhysicalOrg() {
    // 项目启动时把体检机构加载到Redis缓存
