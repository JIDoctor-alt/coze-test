## 一、前置知识：什么是定时任务？

### 1.1 生活中的"定时任务"比喻 🌟

想象一下你每天早上7点起床的场景：

| 场景 | 对应概念 | 生活中的例子 |
|------|---------|-------------|
| **手机闹钟** | **XXL-Job调度中心** | 统一管理所有起床、开会、吃药提醒 |
| **给XXL-Job注册一个任务** | **@XxlJob注解** | 在闹钟App里新建一个"7点起床"的任务 |
| **手机自带定时器** | **@Scheduled注解** | 用手机自带闹钟定个时 |
| **Cron表达式** | **闹钟的"高级设置"** | "每天早上7点，每隔1小时响一次" |

### 1.2 两种定时任务方案对比

| 方案 | 比喻 | 特点 | 适用场景 |
|------|-----|------|---------|
| **XXL-Job调度中心** | 专业闹钟管家 | 统一管理、可视化界面、失败重试、任务分片、任务依赖 | 生产环境、复杂任务、需要监控 |
| **Spring @Scheduled** | 手机自带定时器 | 简单、代码内配置、无法统一管理 | 简单定时任务、开发测试 |

**大白话理解**：
- `@Scheduled` = 你自己写的备忘录，只有你知道，关机就没了
- `@XxlJob` = 交给专业的管家（XXL-Job），他帮你记着，还会提醒你、帮你重试

### 1.3 本项目定时任务注册方式

本项目使用的是**XXL-Job调度中心**方案：
- 所有定时任务类都在 `/src/main/java/com/picchealth/module/scheduling/` 目录下
- 任务通过XXL-Job管理平台配置执行时间和参数
- 调度中心地址和配置在Apollo配置中心管理

### 定时任务1：账号失效处理任务

**一句话人话**：想象公司有规定"员工连续6个月不登录系统就自动离职"，这个任务就是每天检查一遍，把符合条件的"僵尸账号"标记为失效。

**Cron表达式**：需在XXL-Job平台配置（通常每天凌晨执行一次）

**所在类**：`com.picchealth.module.scheduling.AccountEnableTask`

**业务背景**：
- 系统安全策略：防止长期不使用的账号被他人盗用
- 手动不行原因：账号量大（可能几万甚至几十万），人工处理不现实

**核心逻辑逐步拆解**：
```
1. 从PrivilegeUserInfo表中查询"应该失效但还没失效"的账号
   → 条件：最后登录时间距离现在超过6个月（或配置的天数）

2. 遍历这些账号，逐个执行：
   a. 把enable字段设为0（失效）
   b. 记录一条操作日志到AccountRecord表
   c. 操作人写"系统管理员"，类型写"1"（特权用户）

3. 从UpOrgUser表查询机构用户中应该失效的账号
   → 条件：类似上面

4. 遍历机构用户，逐个执行：
   a. 把user_account_locked字段设为"T"（锁定）
   b. 记录操作日志，类型写"2"（机构用户）
```

**调用的Service/Mapper**：
- `PrivilegeUserInfoService.getEndLoginAccount()` - 查询要失效的特权用户
- `PrivilegeUserInfoService.save()` - 保存修改
- `AccountRecordService.save()` - 记录操作日志
- `UpOrgUserDao.getEndLoginAccount()` - 查询要失效的机构用户
- `UpOrgUserService.save()` - 保存修改

**异常处理**：
- 单个账号处理失败不影响其他账号（继续处理下一个）
- 异常信息会打印到日志

**小白易懵点**：
- `PrivilegeUserInfo`和`UpOrgUser`是两个不同的用户体系，一个是特权用户（VIP），一个是机构用户
- `enable=0` vs `user_account_locked='T'`：虽然都是"禁用"，但字段名和值不一样，因为是两套不同的用户体系

**如果停了会怎样**：
- 僵尸账号不会被自动清理，可能存在安全隐患
- 用户长期不登录但账号仍然有效，可能被他人尝试登录

### 定时任务2：审批流转结束任务

**一句话人话**：医院有个审批流程，但有时候医生忘记点了，患者的单子就卡在那里。这个任务就是"超时自动审批"，把卡住超过一定时间的单子强行结案。

**Cron表达式**：需在XXL-Job平台配置（通常间隔执行，如每10分钟）

**所在类**：`com.picchealth.module.scheduling.Approval2EndYlTask`

**业务背景**：
- 工作流引擎（Activiti）的兜底处理机制
- 某些特殊情况下，审批节点可能永远等不到处理人操作
- 手动不行原因：需要实时监控，不可能一直盯着

**核心逻辑逐步拆解**：
```
1. 从Activiti工作流表中查询"待处理且应该结束"的审批任务
   → SQL: getApproval2EndYl() - 返回需要强制结束的审批实例

2. 遍历这些审批任务：
   a. 拾取任务（claim）："我来处理这个任务"
   b. 完成任务（complete），并传入type=2（表示强制结束）
   c. 工作流会按配置的路由走完后续流程
```

**调用的Service/Mapper**：
- `ActivitiDao.getApproval2EndYl()` - 查询待结束的审批
- `TaskService.claim()` - 拾取任务
- `TaskService.complete()` - 完成任务

**异常处理**：
- 异常会被捕获并记录到日志
- 单个任务失败不影响其他任务

**小白易懵点**：
- 这是工作流引擎（Activiti）的操作，不是普通数据库操作
- `claim`和`complete`是Activiti的标准API，理解成"签收"和"提交"

**如果停了会怎样**：
- 超时的审批单会一直卡着，影响用户体验
- 患者可能因为审批超时而无法享受医保待遇

### 定时任务3：宝鸡自动备案任务

**一句话人话**：宝鸡65岁以上的老年人，医保备案快到期时，系统自动帮他们续3年，不用老人跑腿。

**Cron表达式**：需在XXL-Job平台配置（通常每天凌晨执行）

**所在类**：`com.picchealth.module.scheduling.BJAutoFilingTask`

**业务背景**：
- 宝鸡市政策：65岁以上居民享受3年自动续期待遇
- 手动不行原因：涉及大量用户，人工处理效率低且容易出错

**核心逻辑逐步拆解**：
```
1. 查询"永久享受自动备案待遇"的宝鸡居民数据
   → 条件：65岁以上，已享受3年待遇

2. 遍历每个用户：
   a. 判断待遇是否在1个月内到期（提前30天续期）
   
   b. 如果快到期了：
      - 在备案明细表(TMbPutOnRecDetl)插入新记录
      - 待遇开始时间 = 原到期日 + 1天
      - 待遇结束时间 = 新开始时间 + 3年
   
   c. 更新永久享受表(TMbPutOnRecDetlPerpetual)的待遇时间
      - 往后延续3年

3. 涉及的地区：宝鸡（UnitConfigEnum.BJ）
```

**调用的Service/Mapper**：
- `TMbPutOnRecDetlPerpetualDao.getExpireData()` - 查询快到期的永久备案数据
- `TMbPutOnRecDetlService.save()` - 保存新的备案记录
- `TMbPutOnRecDetlPerpetualService` - 更新永久备案待遇时间

**异常处理**：
- 异常会被捕获并记录
- 单个用户处理失败继续处理下一个

**小白易懵点**：
- `TMbPutOnRecDetl`是"备案流水明细表"，记录每次备案
- `TMbPutOnRecDetlPerpetual`是"永久享受自动备案待遇表"，记录有自动续期资格的用户

**如果停了会怎样**：
- 65岁以上老人的医保备案到期后不会自动续期
- 老人需要自己跑腿重新办理，影响就医体验

### 定时任务4：宝鸡备案删除任务

**一句话人话**：医保备案有有效期，过期了就删掉，避免系统里堆满过期数据。

**Cron表达式**：需在XXL-Job平台配置（通常每天执行）

**所在类**：`com.picchealth.module.scheduling.BJDelFilingTask`

**业务背景**：
- 数据清理机制，保持数据新鲜度
- 手动不行原因：需要知道哪些备案确实过期了，需要对比医保系统数据

**核心逻辑逐步拆解**：
```
1. 查询"已过期的备案记录"
   → GhiInsureFiling表的结束日期 < 当前日期

2. 对每个过期备案执行：
   a. 删除mbmz表中的对应病种记录
      - 如果用户还有其他病种：只移除该病种
      - 如果用户只有这一个病种：标记整条记录失效
   
   b. 更新GhiInsureFiling表状态为"已处理"(status=1)
   
   c. 删除VipReviewInfo表的复审记录
```

**调用的Service/Mapper**：
- `GhiInsureFilingDao.getEndList()` - 查询过期备案
- `VipAccountmbmzService` - 处理门诊慢病记录
- `GhiInsureDetailService` - 处理病种明细
- `VipReviewInfoService.delete()` - 删除复审记录

**异常处理**：
- 异常会打印日志
- 单个失败不影响整体

**小白易懵点**：
- 一个用户可能有多个病种（糖尿病+高血压），删除备案时不能把整个用户删了
- 需要用逗号分隔存储多个病种代码

**如果停了会怎样**：
- 过期备案数据会一直堆积
- 可能导致用户看到"已过期"的备案还在系统中

### 定时任务5：宝鸡共享库任务

**一句话人话**：宝鸡医保有个"共享库"，医院需要把备案信息和处方信息推送给它。这个任务就是负责推送，同时检测接口是否正常。

**Cron表达式**：需在XXL-Job平台配置（通常间隔执行）

**所在类**：`com.picchealth.module.scheduling.BJGXTTask`

**业务背景**：
- 医保系统互联互通要求
- 需要把本地数据和医保共享库同步

**核心逻辑逐步拆解**：
```
方法1: repeatedRequestsTask() - 重复推送任务
1. 查询"未推送的备案记录"，推送给共享库
2. 查询"未推送的处方记录"，推送给共享库

方法2: detectionSwitchTask() - 接口健康检测
1. 从Redis读取开关状态
2. 如果开关为false（上次检测失败）：
   - 尝试调用共享库接口
   - 如果成功：设置开关为true
   - 如果失败：保持false
3. 如果开关为true：不处理
```

**调用的Service/Mapper**：
- `CallCxcfRecordService.selectAll()` - 查询备案记录
- `CallCxcfService.recordInsertScheduled()` - 推送备案
- `CallCxcfPrescriptionService.selectAll()` - 查询处方记录
- `CallCxcfService.prePoolIncreasesScheduled()` - 推送处方
- `RedisUtil` - 读写开关状态

**异常处理**：
- 接口调用失败会记录日志
- 检测失败会影响开关状态，用于后续处理决策

**小白易懵点**：
- `CallCxcfService`是核心服务，负责调用医保共享库接口
- Redis开关用于"快速失败"机制：如果接口持续失败，关闭相关功能

**如果停了会怎样**：
- 未推送的数据会一直积压
- 接口异常时无法自动恢复，影响医保业务

### 定时任务6：药品同步任务

**一句话人话**：药房上了新药，或者药品信息变了，这个任务负责把药品信息同步到系统中。

**Cron表达式**：需在XXL-Job平台配置

**所在类**：`com.picchealth.module.scheduling.DrugSynchronizedTask`

**业务背景**：
- 药品目录需要和医保系统保持一致
- 手动不行原因：新药上架频率不固定，需要实时同步

**核心逻辑逐步拆解**：
```
1. 查询"新上架或信息变更"的药品
   → VipDrugstoreOrderService.getNewDrugs()

2. 调用同步服务处理这些药品
   → VipDrugstoreProductService.drugSynchronized(drugs)
```

**调用的Service/Mapper**：
- `VipDrugstoreOrderService.getNewDrugs()` - 查询新药品
- `VipDrugstoreProductService.drugSynchronized()` - 同步药品

**异常处理**：
- 异常会被捕获并记录
- 抛出CustomException会导致任务失败重试

**小白易懵点**：
- 药品同步可能涉及多个系统的数据一致性
- 需要关注医保系统的药品编码对应关系

**如果停了会怎样**：
- 新上架的药品无法在系统中使用
- 患者可能无法开出新药

### 定时任务7：文件转图片任务

**一句话人话**：用户上传的Excel处方文件，系统自动转成图片，方便在其他设备上查看。

**Cron表达式**：原代码中配置的是`0 0/10 * * * ?`（每10分钟），实际需在平台配置

**所在类**：`com.picchealth.module.scheduling.File2ImageTask`

**业务背景**：
- Excel文件在手机上查看不方便
- 转成图片后显示效果更好

**核心逻辑逐步拆解**：
```
1. 查询"需要转换的Excel文件"
   → 条件：状态为待转换的文件

2. 遍历每个文件：
   a. 从SFTP服务器下载Excel文件
   b. 调用转换服务把Excel转成图片
   c. 上传图片到SFTP服务器
   d. 更新文件状态
```

**调用的Service/Mapper**：
- `VipMbdeclareFileDao.queryVIPMbdeclareFile()` - 查询待转换文件
- `File2ImageService.file2image()` - 执行文件转换

**异常处理**：
- 单个文件转换失败打印日志并跳过，继续处理下一个

**小白易懵点**：
- 文件存储在SFTP服务器，不是本地磁盘
- Excel转图片可能因为格式问题失败

**如果停了会怎样**：
- 用户上传的Excel文件不会自动转成图片
- 用户在手机上查看处方不方便

### 定时任务8：医保卡绑定初始化任务

**一句话人话**：用户刚绑定了医保卡，这个任务负责给用户开通VIP会员服务，包括发卡、绑定产品等。

**Cron表达式**：需在XXL-Job平台配置

**所在类**：`com.picchealth.module.scheduling.GhiInsureDetailInitTask`

**业务背景**：
- 绑定医保卡后需要一系列初始化操作
- 手动不行原因：流程复杂，涉及多个步骤

**核心逻辑逐步拆解**：
```
1. 处理"需要发卡的"数据（status=0, flowstatus=2）
   → 针对宝鸡慢病（七种病、两病、十种病）
   → 调用initData()执行发卡和开通一卡通会员

2. 处理"需要绑定产品的"数据（status=1, flowstatus=3）
   → 调用bindingProduct()绑定产品

3. 每一步处理结果记录到VipTaskResult表
```

**调用的Service/Mapper**：
- `GhiInsureDetailService.selectPage()` - 分页查询数据
- `GhiInsureDetailInitService.initData()` - 初始化数据（发卡）
- `GhiInsureDetailInitService.bindingProduct()` - 绑定产品
- `VipTaskResultService.save()` - 记录任务结果

**所在类**：`com.picchealth.module.scheduling.VipAccountMoneyResetTask`

**业务背景**：
- 医保账户有年度额度，过期作废
- 新一年需要重置额度

**核心逻辑逐步拆解**：
```
1. 先调用resetPresQty()重置处方数量
   → 把已用数量清零

2. 分页查询"有效且为保额类型"的账户
   → 每页1000条

3. 对每个账户调用resetMbmzData()
   → 重置门诊慢病额度
   → 重置起付线数据
```

**调用的Service/Mapper**：
- `VipAccountMoneyResetService.resetPresQty()` - 重置处方数量
- `VipAccountMoneyResetService.resetMbmzData()` - 重置账户数据
- `VipAccountDao.getAccountMbmzList()` - 查询账户列表

**异常处理**：
- 单个账户失败跳过，继续处理下一个

**小白易懵点**：
- "保额"和"起付线"是医保概念
  - 保额：年度最高报销额度
  - 起付线：超过这个金额才报销

**如果停了会怎样**：
- 用户年度额度不会重置
- 可能影响医保报销

### 定时任务22：自动脱敏任务 ⭐重要任务

**一句话人话**：处方图片上有患者姓名、身份证号等信息，这个任务自动给敏感信息打码，保护隐私。

**Cron表达式**：需在XXL-Job平台配置（通常每小时执行）

**所在类**：`com.picchealth.module.scheduling.VipMbAutoMaskTask`

**业务背景**：
- 处方图片流转需要脱敏
- 保护患者隐私是法规要求

**核心逻辑逐步拆解**：
```
1. 解析入参（declareIDs, filetype）
   → 如果没传入：查询1小时内新增的需要脱敏的数据

2. 创建线程池处理（默认10个线程）

3. 对每个申报ID：
   a. 检查是否正在脱敏中（防止重复处理）
   b. 更新状态为"脱敏中"
   c. 下载图片
   d. 调用AutoMaskThread进行脱敏
   e. 记录结果

4. 关闭线程池
```

**调用的Service/Mapper**：
- `CallMaskDao.queryNeedAutoMaskDeclareIDList()` - 查询待脱敏数据
- `SFTPUtils` - SFTP文件操作
- `VipIntelligentAuditService` - 审核状态更新
- `AutoMaskThread` - 脱敏线程（多线程处理）

**异常处理**：
- 单个文件失败不影响其他
- 异常会被捕获并记录

**小白易懵点**：
- 为什么用多线程？因为图片处理是IO密集型，CPU利用率低，多线程可以提高效率
- declareId是什么？是申报记录的唯一标识

**如果停了会怎样**：
- 新上传的图片不会被脱敏
- 可能违反隐私保护法规

### 定时任务24：慢病信息更新任务

**Cron表达式**：需在XXL-Job平台配置

**所在类**：`com.picchealth.module.scheduling.VipMbmzUpdateTask`

**业务背景**：
- 病种信息可能需要批量修改
- 比如：某个病种的待遇标准调整

**核心逻辑逐步拆解**：
```
1. 分页查询待更新的数据
   → status=NO（未处理）
   → cvalidate=今天
   → dr=0（有效）

```

**调用的Service/Mapper**：

- `VipMbmzUpdateService.mbmzUpdate()` - 执行更新

**异常处理**：
- 单个失败不影响其他
- 错误信息会记录到数据库

**小白易懵点**：
- cvalidate可能是"验证日期"字段，用于控制生效时间

**如果停了会怎样**：
- 待更新的病种信息不会被更新
- 可能影响患者待遇

## 七、常见问题排查指南

### Q1: 某个定时任务失败了怎么办？

**排查步骤**：
1. 登录XXL-Job管理平台，查看任务执行日志
2. 搜索关键词`定时任务`、`定时任务失败`、`任务执行异常`
3. 查看数据库`vip_task_result`表记录

### Q2: 短信发送不出去怎么办？

**排查步骤**：
1. 检查短信发送时间段（08:00-22:00）
2. 检查短信余额
3. 查看`vip_send_message`表sendflag字段（1=已发送，2=待发送，3=发送失败）
4. 联系短信服务商

### Q3: 医保备案失败怎么办？

**排查步骤**：
1. 查看`t_mb_put_on_rec_error`表的错误信息
2. 确认医保接口是否正常（联系医保局）
3. 检查入参数据是否完整

### Q4: 图片处理卡住了怎么办？

**排查步骤**：
1. 检查SFTP服务器连接
2. 查看`vip_intelligent_audit`表状态
3. 检查AI服务接口是否正常
