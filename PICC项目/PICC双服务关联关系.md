## 一、先认识两个"服务员"

想象一下，你去一家大型餐厅吃饭。这家餐厅有两个超级重要的服务员：

| 服务员 | 花名（项目名） | 工牌号（端口） | 职责 |
|--------|---------------|---------------|------|
| **权限管家** | picc-mzmtb-user | 9092 | 负责"你是不是员工"、"你能进哪些门"、"你有啥权限" |
| **业务达人** | picc-mzmtb-server | 9091 | 负责"慢特病怎么申报"、"处方怎么开"、"卡怎么发" |

### 权限管家（picc-mzmtb-user）
```
🎯 核心能力：用户登录、权限管理、机构管理、角色管理、菜单管理
🔧 技术栈：Spring Boot + MyBatis + GaussDB + Redis + Apollo
📊 数据规模：17张业务表
🎒 小弟数量：权限服务规模（相对较小）
```

### 业务达人（picc-mzmtb-server）
```
🎯 核心能力：慢特病申报、审批、发卡、处方、药店、费用管理
🔧 技术栈：Spring Boot + MyBatis + Activiti6 + GaussDB + Redis + Apollo
📊 数据规模：213个Mapper、260+个PO实体、894个API接口
🎒 小弟数量：是权限服务的19倍！（业务服务规模巨大）
```

## 二、架构关系：两个服务是怎么住在一起的？

### 2.1 部署关系（K8s中各过各的）

```
┌─────────────────────────────────────────────────────────────────────┐
│                          K8s 集群（相当于一个大型写字楼）              │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   ┌─────────────────┐              ┌─────────────────┐              │
│   │   Pod: user     │              │   Pod: server   │              │
│   │ ┌─────────────┐ │              │ ┌─────────────┐ │              │
│   │ │   权限管家   │ │              │ │   业务达人   │ │              │
│   │ │   9092端口  │ │              │ │   9091端口  │ │              │
│   │ └─────────────┘ │              │ └─────────────┘ │              │
│   └────────┬────────┘              └────────┬────────┘              │
│            │                                │                       │
└────────────┼────────────────────────────────┼───────────────────────┘
             │                                │
             │  ← 各自独立部署，互不依赖Pod重启  │
```

**特点**：
- ✅ **独立Pod**：两个服务各自住在自己的"单间"里，一个挂了不影响另一个
- ✅ **独立Deployment**：各自的K8s部署配置分开管理
- ✅ **独立副本数**：业务服务可能跑3个副本，权限服务可能跑2个副本

### 2.2 前端怎么找到两个服务？

```
                    ┌─────────────────────────────────┐
                    │         用户浏览器 (Vue前端)      │
                    │   管理后台 + 业务办理 + 移动端     │
                    └─────────────────┬───────────────┘
                                      │
                          ┌───────────▼───────────┐
                          │   Nginx / K8s Ingress  │
                          │     (门卫大爷)          │
                          │   根据URL路由分发请求    │
                          └───────────┬───────────┘
                                      │
              ┌───────────────────────┼───────────────────────┐
              │                       │                       │
              │  路由规则示例：        │                       │
              │  /user/*  → 9092     │  /api/*  → 9091       │
              │  /privilege/* → 9092 │  /mb/*  → 9091        │
              │                       │                       │
              ▼                       ▼                       ▼
    ┌─────────────────┐      ┌─────────────────┐
    │   权限管家 9092  │      │   业务达人 9091  │
    │   (权限服务)     │      │   (业务服务)     │
    └─────────────────┘      └─────────────────┘
```

**前端调用方式**：
- 🎯 **直接调用**：前端根据功能模块，直接请求对应的服务端口
- 🎯 **网关路由**：Nginx根据URL路径，把请求分发到对应服务
- 🎯 **无BFF层**：业务服务没有专门的网关聚合，直接暴露给前端

### 2.3 共享基础设施

```
┌────────────────────────────────────────────────────────────────────┐
│                      共享资源（中央空调、食堂）                       │
├────────────────────────────────────────────────────────────────────┤
│                                                                    │
│   ┌──────────────┐  ┌──────────────┐  ┌──────────────┐            │
│   │   Redis集群  │  │   Apollo    │  │   GaussDB   │            │
│   │   缓存/会话   │  │   配置中心   │  │   数据库     │            │
│   └──────┬───────┘  └──────┬───────┘  └──────┬───────┘            │
│          │                 │                 │                    │
└──────────┼─────────────────┼─────────────────┼────────────────────┘
           │                 │                 │
           ▼                 ▼                 ▼
    ┌─────────────────────────────────────────────────────┐
    │                    两个服务共享                       │
    │  • Redis：Token缓存、会话管理                        │
    │  • Apollo：配置管理（数据源、接口地址等）              │
    │  • GaussDB：各自独立的Schema/表空间                   │
    └─────────────────────────────────────────────────────┘
```

**共享说明**：
| 资源 | 共享方式 | 说明 |
|------|---------|------|
| **Redis** | ✅ 共享实例 | Token验证、Session缓存、接口限流 |
| **Apollo** | ✅ 共享实例 | 两边都读取同一个Apollo配置空间 |
| **GaussDB** | ⚠️ 独立Schema | 权限库 + 业务库，可能在同一台机器但不同Schema |

## 三、调用关系：谁找谁办事？

### 3.1 核心结论：业务服务主动找权限服务

```
业务达人（9091） ──────────Feign调用────────▶ 权限管家（9092）
         │                                           ▲
         │                                           │
         │        "喂，用户要申报慢特病，               │
         │         帮我查查他有权限吗？"                 │
         │                                           │
         └───────────────────────────────────────────┘
                    "OK，他有权限，让他办"
```

### 3.2 业务服务调用权限服务的场景

| 场景 | 业务服务问权限管家 | 权限管家回答 |
|------|------------------|------------|
| 🔐 **用户登录** | "这个token有效吗？" | 有效/无效 |
| 👤 **获取用户信息** | "张三的详细信息？" | 用户名、机构、状态 |
| 🏢 **获取机构信息** | "张三属于哪个机构？" | 省→市→县 树形结构 |
| 🎭 **获取角色信息** | "张三有哪些角色？" | 管理员/审核员/经办人 |
| 📋 **获取菜单权限** | "张三能看到哪些菜单？" | 申报管理/处方管理等 |
| ✅ **权限校验** | "张三能访问这个URL吗？" | 能/不能 |

### 3.3 调用方式：Feign声明式调用

**业务服务怎么"打电话"给权限服务？**

```java
// 业务服务中的Feign客户端（类似通讯录）
@FeignClient(name = "picc-mzmtb-user", url = "localhost:9092")
public interface PrivilegeFeignClient {
    
    // 查询用户信息
    @GetMapping("/privilege/user/{userId}")
    UserInfo getUserById(@PathVariable("userId") String userId);
    
    // 验证Token
    @PostMapping("/privilege/auth/checkToken")
    Boolean checkToken(@RequestParam("token") String token);
    
    // 获取用户菜单
    @GetMapping("/privilege/menu/userMenus")
    List<Menu> getUserMenus(@RequestHeader("Authorization") String token);
}
```

**调用示例**：
```java
@Service
public class MbDeclareService {
    
    @Autowired
    private PrivilegeFeignClient privilegeFeignClient;
    
    public void createDeclare(DeclareRequest request) {
        // 1. 先问权限管家：这个用户有权限吗？
        Boolean hasPermission = privilegeFeignClient.checkToken(request.getToken());
        
        if (!hasPermission) {
            throw new BizException("您没有权限进行此操作");
        }
        
        // 2. 再获取用户信息
        UserInfo user = privilegeFeignClient.getUserById(request.getUserId());
        
        // 3. 业务逻辑：创建申报记录
        // ...
    }
}
```

### 3.4 权限服务是否调用业务服务？

```
┌─────────────────┐      ❌ 不调用       ┌─────────────────┐
│   权限管家 9092  │ <────────────────  │   业务达人 9091  │
│   (只管权限)     │                     │   (被调用方)     │
└─────────────────┘                     └─────────────────┘
```

**结论**：权限服务**不调用**业务服务！

- 权限服务是一个"纯粹"的权限管理服务
- 它只负责：用户、角色、机构、菜单的管理
- 它不会主动去调用业务服务的任何接口
- 完全是"单向调用"关系

```
用户点击"申报慢特病"按钮
          │
          ▼
┌─────────────────────────────────────────────────────────────────┐

```

```
│   权限管家 9092  │                    │   业务达人 9091 │
│   (权限数据库)   │                    │   (业务数据库)  │
└────────┬────────┘                    └────────┬────────┘
         │                                     │
    ┌────▼────┐                          ┌────▼────┐
    │张三创建  │                          │张三申报  │

    │机构ID    │                          │关联字段  │
```

**关键点**：
- ✅ **不复制数据**：业务服务不复制用户表到自己的库
- ✅ **实时查询**：需要用户信息时，通过Feign调用权限服务
- ✅ **共享Redis Token**：Token验证共用Redis

## 六、典型场景调用链路

### 场景1：用户登录 → 申报慢特病

```
┌──────────────────────────────────────────────────────────────────────────────┐
│                         完整调用链路                                         │
└──────────────────────────────────────────────────────────────────────────────┘

[用户]
  │
  │ 1. 打开浏览器，访问登录页
  ▼
[前端] Vue管理后台
  │
  │ 2. 输入用户名密码，点击登录
  ▼
[权限服务] picc-mzmtb-user:9092
  │
  │ 3. 验证用户名密码（查GaussDB的privilege_user_info表）
  │ 4. 生成Token，写入Redis
  │ 5. 返回Token + 用户信息给前端
  ▼
[前端]
  │
  │ 6. 带着Token访问"申报慢特病"页面
  ▼
[业务服务] picc-mzmtb-server:9091
  │
  │ 7. TokenInterceptor拦截请求
  │ 8. 从Redis验证Token
  │ 9. 【Feign调用】获取用户详情
  │    GET /privilege/user/xxx
  │10. 【Feign调用】获取用户菜单权限
  │    GET /privilege/menu/userMenus
  │11. 执行业务：查询用户已有的申报记录
  ▼
[前端] 展示申报列表页面
  │
  │ 12. 点击"新建申报"
  ▼
[业务服务]
  │
  │13. 【Feign调用】验证用户是否有申报权限
  │14. 创建申报记录（INSERT VIP_MBDECLARE_INFO）
  │15. 启动Activiti工作流（创建审批流程实例）
  │16. 返回申报ID给前端
  ▼
[前端] 提示"申报创建成功，等待审核"

┌──────────────────────────────────────────────────────────────────────────────┐
│                         响应时间预算                                         │
├──────────────────────────────────────────────────────────────────────────────┤
│  步骤1-5（登录）: 约 500ms                                               │
```

## 七、常见问题FAQ

### Q1：为什么业务服务要调用权限服务？
> **答**：因为"用户管理"是基础设施，每个业务都要用到。如果每个业务都自己管理用户，会出现：
> - 用户在申报系统改了密码，处方系统不知道 → 数据不一致
> - 每个系统都要写用户管理的代码 → 重复劳动
> - 权限控制逻辑分散 → 安全漏洞

### Q2：权限服务挂了，业务还能用吗？
> **答**：**部分能用，部分不能用**
> - ✅ 已登录用户的**基本操作**：可能正常（如果Token还在Redis中）
> - ❌ **新登录**：不行（登录接口在权限服务）
> - ❌ **权限校验**：不行（会Feign超时）
> - ❌ **获取用户信息**：不行（会报错）

### Q3：两个服务用同一个Redis，会有冲突吗？
> **答**：**不会**
> - Redis Key有前缀区分：`token:*`、`user:*`、`cache:*`
> - 不同服务的Key不会互相覆盖
> - 但如果Redis挂了，**两个服务都会受影响**

### Q4：数据库是共享的吗？
> **答**：**逻辑上分离**
> - 权限库 + 业务库可能在同一台GaussDB
> - 但Schema（命名空间）分开
> - 业务服务**不直接查权限库的表**，而是通过Feign调用

### Q5：前端怎么知道请求哪个服务？
> **答**：**Nginx根据URL路由**
> - `/user/*` → 权限服务
> - `/api/*` → 业务服务
> - 前端代码写死了对应的接口地址
